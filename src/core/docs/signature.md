# Binary Signature Design

## 1. 概要

mnuカーネルおよびmochiOSでは、バイナリ署名を用いて、カーネルモジュール、システムサービス、アプリケーションなどの実行ファイルが改ざんされていないことを確認します。

署名データベースは、登録された公開鍵、バイナリの署名情報、失効情報、要求Capabilityなどを管理します。

署名データベースの登録、更新、照会、失効、署名検証は、カーネルのシステムコールとして実装せず、ユーザー空間で動作する署名管理サービスへのIPCとして提供します。

カーネルは署名形式やデータベース形式を解釈せず、Capabilityの管理、IPCの強制、プロセス生成、メモリ保護などの基本機構のみを担当します。

## 2. 設計方針

署名機能は、次の方針に基づいて設計します。

- 署名データベースはユーザー空間のサービスが管理します
- 署名データベースへの操作はIPCを使用します
- カーネルに署名専用システムコールは追加しません
- 署名検証結果だけでCapabilityを無条件に付与しません
- 実行するバイナリと検証したバイナリが同一であることを保証します
- 公開鍵、署名、Capability要求を一体として検証します
- 鍵および署名の失効をサポートします
- 署名サービス自身は初期信頼基盤として扱います

この構成は、mnuの「Policyはユーザー空間、Enforcementはカーネル」という設計方針に従います。

署名の信頼判断、公開鍵の管理、失効規則、Capability付与方針などはPolicyに該当します。

一方、Capabilityを持たないプロセスの操作拒否、メモリ保護、IPCの送信元保証、プロセス隔離などはEnforcementに該当し、カーネルが担当します。

## 3. システム構成

署名機能は、主に次のコンポーネントから構成されます。

### 3.1 署名管理サービス

署名管理サービスは、署名データベースを管理するユーザー空間サービスです。実際のサービス名は、ユーザーランドの実装によって変わります。

主な責務は次のとおりです。

- 公開鍵の登録
- 公開鍵の照会
- 公開鍵の失効
- バイナリ署名情報の登録
- バイナリ署名情報の照会
- バイナリ署名情報の失効
- バイナリ署名の検証
- Capability要求情報の返却
- データベースの永続化
- データベースの整合性確認
- 監査情報の記録

署名管理サービスは、署名対象のバイナリを実行しません。

署名検証と信頼情報の管理だけを担当します。

これは、core.serviceから呼び出されるサービスです。（mnuはcore.serviceのみを直接呼び出します）

### 3.2 process.service

`process.service`は、実行ファイルの読み込みとプロセス生成を管理します。

主な責務は次のとおりです。

- 実行ファイルの読み込み
- manifestの読み込み
- バイナリハッシュの計算
- manifestハッシュの計算
- 署名管理サービスへの検証要求
- 要求Capabilityの評価
- 子プロセスへ渡すCapabilityの生成
- `process_spawn`の呼び出し
- 起動失敗時のリソース解放

`process.service`は、署名データベースを直接読み書きしません。

署名情報の取得と検証は、必ず署名管理サービスへのIPCを通して行います。

### 3.3 管理ツール

管理ツールは、署名データベースの登録や失効を行うユーザー空間プログラムです。

主な操作は次のとおりです。

- 公開鍵の登録
- 公開鍵の失効
- バイナリ署名の登録
- バイナリ署名の失効
- 登録情報の照会

管理ツールはデータベースファイルを直接変更してはいけません。

すべての操作は署名管理サービスへのIPCとして実行します。

### 3.4 mnuカーネル

カーネルは署名の暗号学的検証を行いません。

署名機能に関連して、カーネルは次の機構を提供します。

- IPC
- Capabilityの保持と検査
- Capabilityの複製
- Capabilityの制限
- Capabilityの移譲
- プロセス生成
- メモリ割り当て
- メモリ保護
- プロセス間の分離
- ユーザーポインタの検証

署名サービスの侵害を防ぐため、カーネルはIPC送信元とCapabilityの所有関係を保証する必要があります。

## 4. IPCを使用する理由

署名データベースの操作は、システムコールではなくIPCとして実装します。

署名データベースは、カーネルが直接管理すべき基本資源ではありません。

署名方式、公開鍵形式、証明書形式、失効規則、データベース形式などは、将来的に変更される可能性があります。

これらをシステムコールABIに含めると、次の問題が発生します。

- 署名形式の変更がカーネルABIの変更になります
- データベース形式がカーネル実装に依存します
- カーネルの攻撃対象領域が拡大します
- 永続化処理がカーネル内に入り込みます
- 暗号ライブラリをカーネル内に導入する必要があります
- 署名方式の追加や廃止が困難になります
- データベース破損時の復旧処理までカーネルが担当することになります

署名処理をIPCサービスとして分離することで、カーネルABIを変更せずに署名方式やデータベース形式を更新できます。

## 5. Capability

署名サービスへの操作は、Capabilityによって制御します。

少なくとも次のCapabilityを定義します。

| Capability             | 説明              |
|------------------------|-----------------|
| `signature.verify`     | 署名検証を要求できます     |
| `signature.db.read`    | バイナリ署名情報を照会できます |
| `signature.db.write`   | バイナリ署名情報を登録できます |
| `signature.db.revoke`  | バイナリ署名情報を失効できます |
| `signature.key.read`   | 公開鍵情報を照会できます    |
| `signature.key.write`  | 公開鍵を登録できます      |
| `signature.key.revoke` | 公開鍵を失効できます      |

登録権限と失効権限は分離します。

新しい署名情報を追加できるプロセスが、既存の署名情報を自由に失効できる設計は避けます。

また、公開鍵の登録権限と公開鍵の失効権限も分離します。

IPC要求に管理者フラグなどを含めて権限判定してはいけません。

要求メッセージ内の値は送信元が自由に設定できるため、信頼できません。

権限判定には、カーネルが管理するCapabilityのみを使用します。

## 6. IPC操作

署名管理サービスは、次の操作を提供します。

### 6.1 署名検証

署名検証操作は、バイナリとmanifestが登録情報に一致するか確認します。

要求には、次の情報を含めます。

- バイナリハッシュ
- manifestハッシュ
- パッケージID
- 実行ファイル名
- 署名形式のバージョン

成功時には、次の情報を返します。

- 署名者の鍵ID
- 登録されたCapability要求
- 署名レコードの識別情報
- データベースの世代番号
- 検証結果

### 6.2 バイナリ署名情報の登録

登録要求には、次の情報を含めます。

- 署名対象データ
- 公開鍵の識別子
- 署名値
- バイナリハッシュ
- manifestハッシュ
- パッケージID
- 実行ファイル名
- バージョン
- Capability要求

登録時には、署名が有効であることを検証します。

署名検証に失敗した情報をデータベースへ保存してはいけません。

### 6.3 バイナリ署名情報の照会

照会は、バイナリハッシュ、パッケージID、鍵IDなどを条件として実行します。

照会結果には、次の情報を含めます。

- 登録状態
- 失効状態
- 署名者
- 登録日時
- 失効日時
- Capability要求
- バージョン情報

### 6.4 バイナリ署名情報の失効

失効要求には、対象となる署名情報の識別子と失効理由を含めます。

失効した署名は削除しません。

失効状態としてデータベースに残します。

これにより、過去に存在した署名情報と失効履歴を追跡できます。

### 6.5 公開鍵の登録

公開鍵登録では、次の情報を受け取ります。

- 署名アルゴリズム
- 公開鍵データ
- 所有者識別子
- 表示名
- 任意の付加情報

登録時には、公開鍵データから鍵IDを計算します。

同一の鍵IDがすでに存在する場合は、既存レコードを上書きせず、重複エラーを返します。

### 6.6 公開鍵の照会

鍵IDを指定して、次の情報を取得します。

- 公開鍵データ
- 署名アルゴリズム
- 所有者識別子
- 登録日時
- 失効状態
- 失効日時

### 6.7 公開鍵の失効

公開鍵を失効した場合、その鍵で署名されたすべてのバイナリ署名を無効として扱います。

個々の署名レコードをすべて書き換える必要はありません。

署名レコードと公開鍵レコードの両方が有効な場合にのみ、署名を有効と判定します。

## 7. IPCメッセージ形式

すべてのIPC要求には、共通ヘッダーを持たせます。

共通ヘッダーには、次の情報を含めます。

- プロトコルバージョン
- 操作番号
- 要求ID
- ペイロード長
- フラグ

応答には、次の情報を含めます。

- プロトコルバージョン
- 操作番号
- 要求ID
- ステータスコード
- ペイロード長

要求IDは、要求と応答の対応確認に使用します。

プロトコルバージョンが未対応の場合は、処理を行わずエラーを返します。

ペイロード長は受信したメッセージサイズと一致していることを確認します。

操作ごとに最大メッセージサイズを定め、過大な要求を拒否します。

## 8. 署名アルゴリズム

初期実装では、Ed25519のみをサポートします。

署名アルゴリズムを識別する列挙値は定義しますが、未対応のアルゴリズムは拒否します。

初期段階から複数の署名方式を実装すると、次の処理が複雑になります。

- 公開鍵長の検証
- 署名長の検証
- 鍵IDの生成
- エラー処理
- DB形式
- 署名対象の正規化

将来的な拡張可能性を残しつつ、最初の実装はEd25519に限定します。

## 9. 鍵ID

公開鍵の識別には、表示名や所有者名を使用しません。

鍵IDは、公開鍵データから計算した暗号学的ハッシュとします。

表示名や所有者名は変更される可能性がありますが、公開鍵データから計算された鍵IDは安定しています。

鍵IDの計算にはSHA-256を使用します。

同一の公開鍵データからは、常に同一の鍵IDが生成されます。

## 10. 署名対象

署名対象には、バイナリ本体だけでなく、次の情報を含めます。

- 署名形式のバージョン
- バイナリハッシュ
- manifestハッシュ
- パッケージID
- 実行ファイル名
- バージョン
- 要求Capability

バイナリだけを署名対象にすると、manifestやCapability要求を書き換えられる可能性があります。

そのため、実行に影響する情報をまとめて署名対象とします。

署名対象データは、必ず正規化されたバイナリ形式に変換します。

TOMLやJSONの文字列表現をそのまま署名対象としてはいけません。

TOMLやJSONは、空白、改行、キーの順序、エスケープ形式などによって、意味が同じでもバイト列が変化するためです。

正規化形式では、次の仕様を固定します。

- フィールド順序
- 整数のエンディアン
- 文字列の文字コード
- 文字列長の表現
- 配列要素の順序
- 真偽値の表現
- Capabilityの並び順

文字列はUTF-8とします。

整数はlittle-endianに統一します。

Capabilityの順序によって署名結果が変わらないようにする場合は、署名前にCapabilityを名前、リソース、権利の順で並べ替えます。

## 11. Capability要求

署名情報には、バイナリが要求するCapabilityを含めます。

Capability要求には、少なくとも次の情報を持たせます。

- Capability名
- 対象リソース
- 権利ビット
- 任意の制約情報

署名に含まれるCapability要求は、付与を保証するものではありません。

実際に付与するCapabilityは、次の条件をすべて満たす必要があります。

1. 署名されたmanifestで要求されていること
2. システムポリシーで許可されていること
3. 起動元が委譲可能なCapabilityであること
4. ユーザー設定によって許可されていること
5. 失効や制限の対象になっていないこと

したがって、実際の付与Capabilityは、要求Capability、システムポリシー、起動元の保有Capabilityの共通部分となります。

署名済みであることだけを理由に、カーネルが新しいCapabilityを生成してはいけません。

## 12. データモデル

署名データベースでは、少なくとも次の種類のレコードを管理します。

### 12.1 公開鍵レコード

公開鍵レコードは、次の情報を保持します。

- 鍵ID
- 署名アルゴリズム
- 公開鍵データ
- 所有者識別子
- 表示名
- 登録日時
- 状態
- 失効日時
- 失効理由

状態には、少なくとも次の値を持たせます。

- 有効
- 失効済み

削除状態を追加する場合でも、監査目的の情報は別途保持します。

### 12.2 バイナリ署名レコード

バイナリ署名レコードは、次の情報を保持します。

- バイナリハッシュ
- manifestハッシュ
- 鍵ID
- 署名値
- パッケージID
- 実行ファイル名
- バージョン
- Capability要求
- 登録日時
- 状態
- 失効日時
- 失効理由

同じバイナリに複数の署名が存在する可能性を考慮し、バイナリハッシュだけを一意キーにしません。

少なくとも、バイナリハッシュ、manifestハッシュ、鍵IDの組み合わせで識別できるようにします。

### 12.3 監査レコード

管理操作には、監査情報を残します。

監査レコードには、次の情報を含めます。

- 操作種別
- 対象レコード
- 操作元プロセス
- 使用されたCapability
- 操作日時
- 成功または失敗
- 失敗理由
- 失効理由
- データベース世代番号

秘密鍵や署名対象の生データを監査ログへ記録してはいけません。

## 13. データベースの永続化

署名データベースは、ビルド時に生成して rootfs へ配置します。実装例では `signature.db` を使い、起動時にカーネルがそれを読み込みます。

初期構成では、次のように分割できます。

- 公開鍵データベース
- バイナリ署名データベース
- 監査ログ
- 世代番号
- バックアップデータベース

データベースファイルを直接上書きすると、書き込み途中の停止やクラッシュによって破損する可能性があります。

更新時には、一時ファイルを使用します。

更新手順は次のとおりです。

1. 現在のデータベースをメモリ上で更新します
2. 新しい内容を一時ファイルへ書き込みます
3. 一時ファイルを同期します
4. 現在のデータベースをバックアップへ変更します
5. 一時ファイルを正式なデータベース名へ変更します
6. 正式ファイルを同期します
7. 世代番号を更新します

起動時には、正式なデータベースを読み込みます。

正式なデータベースが破損している場合は、バックアップを読み込みます。

両方が読み込めない場合は、署名サービスの起動を失敗させます。

破損したデータベースを空のデータベースとして扱ってはいけません。

空のデータベースとして起動すると、本来拒否されるべきバイナリが誤って許可される可能性があります。

## 14. データベースの整合性

各データベースファイルには、共通ヘッダーを持たせます。

ヘッダーには、次の情報を含めます。

- マジック値
- データベース形式のバージョン
- 世代番号
- レコード数
- ペイロード長
- ペイロードハッシュ

読み込み時には、次の項目を確認します。

- マジック値が一致していること
- 形式バージョンが対応範囲内であること
- ファイルサイズが宣言されたサイズと一致すること
- レコード数が上限以下であること
- ペイロードハッシュが一致すること
- 各レコードの長さが有効であること
- 列挙値が定義済みであること
- 文字列が有効なUTF-8であること

ペイロードハッシュは、ストレージ破損の検出に使用します。

データベース自体の真正性が必要な場合は、データベース全体への署名も追加します。

ただし、各バイナリ署名レコードについては、ロード時に公開鍵を用いて再検証します。

## 15. 署名登録処理

バイナリ署名情報の登録は、次の手順で行います。

1. IPCメッセージの形式を確認します
2. プロトコルバージョンを確認します
3. ペイロード長を確認します
4. 送信元の`signature.db.write` Capabilityを確認します
5. 署名対象データをデコードします
6. 各フィールドの長さと値を検証します
7. 鍵IDから公開鍵を検索します
8. 公開鍵が有効であることを確認します
9. 署名アルゴリズムを確認します
10. 署名対象データを正規化します
11. 署名を検証します
12. 重複する署名情報が存在しないことを確認します
13. データベースへ登録します
14. データベースを永続化します
15. 監査ログを記録します
16. 成功を返します

署名検証に失敗した場合は、登録処理を行いません。

データベースへの保存に失敗した場合は、メモリ上の変更も取り消します。

## 16. 署名検証処理

署名検証は、次の手順で行います。

1. IPC要求の形式を確認します
2. 送信元の`signature.verify` Capabilityを確認します
3. バイナリハッシュに対応する署名レコードを検索します
4. manifestハッシュが一致することを確認します
5. パッケージIDが一致することを確認します
6. 実行ファイル名が一致することを確認します
7. 署名レコードが失効していないことを確認します
8. 公開鍵が失効していないことを確認します
9. 署名対象データを再構築します
10. 公開鍵で署名を再検証します
11. Capability要求を取得します
12. 検証結果を返します

データベース登録時に署名を検証していても、実行時に再検証します。

これにより、データベースファイルの破損や不正な変更を検出できます。

検証結果は一定期間キャッシュできますが、次の場合はキャッシュを無効化します。

- 公開鍵が失効した場合
- 署名レコードが失効した場合
- データベース世代番号が変化した場合
- システムポリシーが変更された場合

## 17. プロセス起動処理

プロセス起動は、次の手順で行います。

1. `process.service`が実行ファイルを開きます
2. ファイルサイズを取得します
3. サイズの上限を確認します
4. 実行ファイルをメモリへ読み込みます
5. バイナリハッシュを計算します
6. manifestを読み込みます
7. manifestハッシュを計算します
8. 実行ファイルのメモリを読み取り専用にします
9. 署名管理サービスへ検証要求を送信します
10. 署名検証結果を受け取ります
11. 要求Capabilityをシステムポリシーと照合します
12. 起動元が委譲可能なCapabilityと照合します
13. 子プロセス用Capabilityを制限付きで複製します
14. 読み込み済み実行イメージを使用して`process_spawn`を呼び出します
15. 子プロセスへCapabilityを移譲します
16. 起動に使用した一時リソースを解放します

署名検証に失敗した場合、`process_spawn`を呼び出してはいけません。

## 18. TOCTOU対策

署名検証したファイルと、実際に実行するファイルが異なる問題を防ぐ必要があります。

次の流れは使用してはいけません。

1. パスからファイルを開く
2. 署名を検証する
3. ファイルを閉じる
4. 同じパスから再度ファイルを開く
5. 実行する

検証後から再オープンまでの間に、別のバイナリへ置き換えられる可能性があります。

正しい処理では、実行ファイルを一度だけ読み込みます。

読み込んだ同一のメモリイメージに対してハッシュを計算し、その同じメモリイメージをプロセス生成に使用します。

検証後にファイルパスから再読み込みしてはいけません。

より厳密な実装では、検証対象のメモリを変更不能なメモリオブジェクトとして扱います。

初期実装では、次の条件を満たせば成立します。

- `process.service`を信頼されたサービスとして扱うこと
- ハッシュ計算後に対象メモリを書き込み不可とすること
- `process_spawn`が対象イメージをカーネル内部へコピーすること
- プロセス生成中に元のバッファを書き換えられないこと

将来的には、メモリオブジェクトを恒久的に変更不能とする汎用的なseal機構を追加できます。

ただし、これは署名専用機能ではなく、メモリ管理機能として設計します。

## 19. process_spawnの要件

`process_spawn`は、署名を解釈しません。

ただし、安全なプロセス生成のため、次の条件を満たす必要があります。

- 呼び出し元にプロセス生成Capabilityが必要です
- 実行イメージのアドレス範囲を検証します
- 実行イメージ全体を安全にコピーします
- ELFヘッダーとセグメントを検証します
- ユーザー空間範囲外へのロードを拒否します
- 既存のマッピングとの重複を拒否します
- 書き込み可能かつ実行可能なページを原則として拒否します
- 子へ渡すCapabilityが呼び出し元の権限以下であることを確認します
- 失敗時に割り当て済みページを解放します
- 失敗時に生成途中のCapabilityを破棄します
- 失敗時に不完全なプロセスを残しません

`process_spawn`がパスを受け取って内部でファイルを再度開く設計は避けます。

実行イメージまたはメモリオブジェクトを受け取る設計にします。

## 20. ブートストラップ

署名管理サービスを通常の署名検証経路で起動しようとすると、署名管理サービス自身が必要になるため循環が発生します。

この問題を避けるため、次のコンポーネントを初期信頼基盤として扱います。

- カーネル
- 初期プロセス
- 署名管理サービス
- `process.service`
- 初期公開鍵データベース
- 初期Capabilityポリシー

これらは、信頼されたinitfsまたはブートイメージに含めます。

起動順序は次のとおりです。

1. ブートローダーがカーネルとinitfsを読み込みます
2. カーネルが初期プロセスを起動します
3. 初期プロセスが署名管理サービスを起動します
4. 署名管理サービスが初期公開鍵を読み込みます
5. 永続データベースを読み込みます
6. データベースの整合性を確認します
7. `process.service`を起動します
8. 以降のサービスを署名検証付きで起動します
9. 通常アプリケーションの起動受付を開始します

初期信頼基盤に含まれるファイルは、通常アプリケーションより厳格に管理します。

可能であれば、initfs全体またはブートイメージ全体に署名を付与します。

## 21. 失効処理

署名の失効には、次の2種類があります。

### 21.1 個別署名の失効

特定のバイナリ署名だけを無効化します。

同じ公開鍵による他の署名は引き続き有効です。

主な用途は次のとおりです。

- 特定バージョンに脆弱性が存在する場合
- 誤って公開したバイナリを停止する場合
- 特定のパッケージだけを無効化する場合

### 21.2 公開鍵の失効

公開鍵自体を無効化します。

その鍵によって署名されたすべてのバイナリを無効として扱います。

主な用途は次のとおりです。

- 秘密鍵が漏えいした場合
- 開発者アカウントが侵害された場合
- 鍵の使用を終了する場合

失効は物理削除ではなく、状態変更として記録します。

失効操作には、理由と操作日時を必須とします。

## 22. エラー処理

署名サービスは、少なくとも次のエラーを区別します。

- メッセージ形式が不正
- プロトコルバージョンが未対応
- 操作番号が不正
- ペイロード長が不正
- 権限が不足
- 対象が存在しない
- 対象がすでに存在する
- 公開鍵が不正
- 公開鍵が失効済み
- 署名方式が未対応
- 署名が不正
- バイナリ署名が失効済み
- バイナリハッシュが不一致
- manifestハッシュが不一致
- パッケージIDが不一致
- データベースが破損
- ストレージ操作に失敗
- メモリ不足
- 内部状態が不正

一般アプリケーションに対しては、暗号検証失敗の詳細を返しすぎないようにします。

例えば、署名値の形式不正、公開鍵不一致、署名対象不一致などを、外部には単一の署名検証失敗として返すことができます。

詳細な原因は監査ログへ記録します。

## 23. 制限値

DoS対策として、次の項目に上限を設定します。

- IPCメッセージサイズ
- パッケージIDの長さ
- 実行ファイル名の長さ
- バージョン文字列の長さ
- 表示名の長さ
- Capability数
- Capability名の長さ
- リソース指定の長さ
- 公開鍵データの長さ
- 署名データの長さ
- データベース内レコード数
- 1回の照会で返す件数
- 実行ファイルサイズ
- manifestサイズ

上限値を超える要求は、データを部分的に処理せず拒否します。

## 24. 並行処理

署名検証要求は並行して処理できます。

一方、データベース更新操作は直列化します。

読み取り中に更新が発生する場合は、世代番号を使用して整合性を管理します。

更新処理では、次の順序を守ります。

1. 更新ロックを取得します
2. 現在の世代番号を確認します
3. 更新内容を検証します
4. 新しいデータベースを作成します
5. 永続化します
6. メモリ上のデータベースを切り替えます
7. 世代番号を更新します
8. 更新ロックを解放します

読み取り処理は、処理開始時の世代番号と処理終了時の世代番号が一致することを確認できます。

一致しない場合は、必要に応じて再試行します。

## 25. キャッシュ

署名検証結果は、性能向上のためにキャッシュできます。

キャッシュキーには、次の情報を含めます。

- バイナリハッシュ
- manifestハッシュ
- 鍵ID
- データベース世代番号
- ポリシー世代番号

キャッシュは、次の場合に無効化します。

- 署名レコードが失効した場合
- 公開鍵が失効した場合
- データベースが更新された場合
- Capabilityポリシーが変更された場合
- 署名アルゴリズムの実装が更新された場合

キャッシュされた検証結果を永続的な信頼情報として扱ってはいけません。

## 26. 監査

次の操作は監査対象とします。

- 公開鍵登録
- 公開鍵失効
- バイナリ署名登録
- バイナリ署名失効
- データベース読み込み失敗
- データベース復旧
- 権限不足による操作拒否
- 署名検証失敗
- 不正なメッセージ受信
- 未対応アルゴリズムの使用
- データベース更新失敗

大量の通常検証成功をすべて永続ログへ記録すると、性能や容量に影響します。

検証成功ログについては、システム設定により記録範囲を変更できるようにします。

## 27. セキュリティ上の注意

### 27.1 署名済みであることと安全であることは同じではない

署名は、バイナリの出所と改ざんの有無を確認するためのものです。

署名済みバイナリに脆弱性や悪意ある処理が存在しないことを保証するものではありません。

そのため、署名済みバイナリであっても、Capabilityによる制限が必要です。

### 27.2 署名サービスは高い権限を持つ

署名管理サービスが侵害されると、不正な鍵や署名情報を登録される可能性があります。

そのため、署名サービスには必要最小限のCapabilityだけを付与します。

署名サービス自身に、任意のプロセス生成権限や物理メモリ操作権限を与えてはいけません。

### 27.3 process.serviceも信頼境界に含まれる

`process.service`が検証後のバイナリを置き換えられる場合、署名検証を回避できます。

そのため、`process.service`も初期の信頼基盤に含まれます。

将来的には、検証済みメモリオブジェクトをカーネル管理の変更不能オブジェクトとして扱うことで、信頼範囲を縮小できます。

### 27.4 データベースファイルへの直接アクセスを制限する

署名データベースファイルは、原則として署名管理サービスだけが書き込み可能とします。

他のプロセスがデータベースを直接変更できる場合、IPCのCapability検査を回避できます。

読み取り権限についても必要最小限にします。

## 28. 既存システムコールとの関係

署名管理のために、新しい署名専用システムコールは追加しません。

既存のシステムコールを次の目的で使用します。

| システムコール          | 用途                  |
|------------------|---------------------|
| `ipc_create`     | 署名サービスのIPCエンドポイント作成 |
| `ipc_send`       | 非同期要求の送信            |
| `ipc_recv`       | 署名要求の受信             |
| `ipc_call`       | 署名検証などの同期要求         |
| `ipc_reply`      | 検証結果の返却             |
| `ipc_wait`       | 要求待機                |
| `cap_clone`      | 子プロセス用Capabilityの複製 |
| `cap_restrict`   | Capabilityの権限縮小     |
| `cap_transfer`   | Capabilityの移譲       |
| `cap_query`      | 受信したCapabilityの検査   |
| `memory_alloc`   | バイナリおよびDB読み込み領域の確保  |
| `memory_free`    | 一時領域の解放             |
| `memory_protect` | 検証済みバイナリの書き込み禁止     |
| `process_spawn`  | 検証済みプロセスの生成         |
| `file_open`      | 実行ファイルおよびDBの読み込み    |
| `file_read`      | ファイル内容の読み込み         |
| `file_write`     | DB一時ファイルの書き込み       |
| `file_rename`    | DBのアトミックな切り替え       |
| `file_sync`      | DB更新内容の同期           |
| `time_now`       | 登録日時や失効日時の取得        |

将来的に変更不能なメモリオブジェクトが必要になった場合は、署名専用ではなく汎用メモリ機構として追加します。

## 29. 実装順序

実装は、次の順序で進めます。

1. 署名IPCプロトコルを定義します
2. エラーコードを定義します
3. 公開鍵レコード形式を定義します
4. バイナリ署名レコード形式を定義します
5. データベースファイル形式を定義します
6. 公開鍵データベースの読み書きを実装します
7. 公開鍵登録と照会を実装します
8. Ed25519署名検証を実装します
9. バイナリ署名登録を実装します
10. バイナリ署名検証を実装します
11. 公開鍵と署名の失効を実装します
12. データベースのアトミック更新を実装します
13. バックアップからの復旧を実装します
14. `process.service`からの検証要求を実装します
15. Capability要求の評価を実装します
16. 検証済みイメージからのプロセス生成を実装します
17. 監査ログを実装します
18. キャッシュと世代番号を実装します

## 30. 最終的な責務分離

最終的な責務は、次のように分けます。

| 処理             | 担当                           |
|----------------|------------------------------|
| 公開鍵登録          | 署名管理サービスへのIPC                |
| 公開鍵照会          | 署名管理サービスへのIPC                |
| 公開鍵失効          | 署名管理サービスへのIPC                |
| バイナリ署名登録       | 署名管理サービスへのIPC                |
| バイナリ署名照会       | 署名管理サービスへのIPC                |
| バイナリ署名失効       | 署名管理サービスへのIPC                |
| 署名検証           | 署名管理サービスへのIPC                |
| データベース永続化      | 署名管理サービス                     |
| ELF読み込み        | `process.service`            |
| バイナリハッシュ計算     | `process.service`            |
| manifestハッシュ計算 | `process.service`            |
| Capability付与判断 | `process.service`およびポリシーサービス |
| Capability制限   | mnuカーネル                      |
| Capability移譲   | mnuカーネル                      |
| プロセス生成         | `process_spawn`              |
| プロセス隔離         | mnuカーネル                      |
| メモリ保護          | mnuカーネル                      |

署名データベースの登録や検証をシステムコールとして実装する必要はありません。

署名データベースはユーザー空間のPolicyとして管理し、カーネルはCapabilityとプロセス隔離を強制する構成とします。
