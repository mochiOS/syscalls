# mnu kernel documentation

mnuカーネルはRustで書かれたマイクロカーネルで、セキュリティと速度、モジュール性を重視して設計されています。
カーネルは、プロセス管理、メモリ管理、IPC、割り込み処理、その他セキュリティ機能などの基本的な機能を提供し、ファイルシステムなどの各種機能はユーザー空間で実装することを想定しています。

## 設計方針

mnuカーネルは、できるだけ小さなカーネル本体に責務を集中させすぎないことを基本方針としています。
ファイルシステム、ネットワークスタック、デバイスドライバ、GUIなどの機能は、可能な限りユーザー空間のサービスとして分離されます。

この設計により、各機能の障害がカーネル全体に波及しにくくなります。
たとえば、ファイルシステムサービスやドライバサービスが異常終了した場合でも、カーネルそのものが即座に停止するのではなく、対象サービスの再起動や隔離によって復旧できる構成を目指します。

mnuでは、以下の要素を重視しています。

- カーネル本体の責務を小さく保つこと
- ユーザー空間サービスによる機能分離
- IPCによる高速で安全なプロセス間通信
- Capabilityによる明示的な権限管理
- ドライバやサービスの障害をカーネル全体へ伝播させないこと
- 将来的なマルチプロセッサ対応を見据えた設計
- セキュリティ機構を後付けではなく基礎設計に含めること

## カーネルの責務

mnuカーネルは、OS全体の土台となる低レベルな機能を提供します。
ただし、一般的なモノリシックカーネルのように、ファイルシステムやネットワーク、複雑なドライバ処理をカーネル内に直接持つことは避けます。

カーネルが主に担当する機能は次のとおりです。

- プロセスとスレッドの管理
- 仮想メモリ管理
- ページテーブル管理
- IPCの提供
- 割り込みと例外の処理
- システムコールの入口
- Capabilityの検査
- ユーザー空間サービスの起動補助
- 最低限のデバイス初期化
- カーネル自身の保護機構

カーネルは、ユーザー空間サービスに対して必要なプリミティブを提供します。
各サービスは、それらのプリミティブを組み合わせてファイルシステム、ドライバ、GUI、アプリケーション実行環境などを構築します。

## ユーザー空間サービス

mnuでは、OSの多くの機能をユーザー空間サービスとして実装します。
サービスは通常のユーザープロセスとして動作しますが、Capabilityによって通常アプリケーションよりも広い権限を持つことがあります。

たとえば、ファイルシステムサービスはストレージデバイスへアクセスするCapabilityを持ち、アプリケーションからのファイル操作要求をIPC経由で受け取ります。
アプリケーションは直接ディスクを操作せず、ファイルシステムサービスを通じて安全にアクセスします。

この構成により、次の利点があります。

- カーネル内コードを減らせる
- サービス単位で障害を隔離できる
- 機能の差し替えや更新がしやすい
- 権限をサービスごとに細かく制御できる
- ドライバやファイルシステムの実装ミスがカーネル破壊につながりにくい

一方で、ユーザー空間サービスを多用する設計ではIPC性能が重要になります。
そのため、mnuではIPCをカーネルの中心的な機能として扱い、低レイテンシかつ安全な通信を目指します。

サービスについては、[サービスドキュメント](./services.md)でさらに詳しく説明されています。

## IPC

IPCは、mnuにおけるサービス間通信の基盤です。
アプリケーション、サービス、ドライバはIPCを使って互いに要求と応答をやり取りします。

IPCは単なるメッセージ送信ではなく、OS全体の構造を支える重要な仕組みです。
ファイルの読み書き、プロセス生成、デバイスアクセス、GUIイベント、サービス制御など、多くの操作がIPCを通して行われます。

mnuのIPCでは、次の性質を重視します。

- 小さなメッセージを高速に送受信できること
- 大きなデータ転送では不要なコピーを減らすこと
- 送信元と送信先を明確に識別できること
- Capabilityと連携して権限を検査できること
- 不正なメッセージや壊れた通信状態を隔離できること

ユーザー空間サービスを中心にしたOSでは、IPCが遅いとOS全体が遅くなります。
そのため、mnuではIPCの設計を後回しにせず、カーネルの中核機能として扱います。

IPCについては、[IPCドキュメント](./ipc.md)でさらに詳しく説明されています。

## Capability

mnuでは、権限管理にCapabilityベースの考え方を取り入れます。
Capabilityは、プロセスやサービスが何を行えるかを表す明示的な権限です。

たとえば、あるサービスがストレージデバイスにアクセスするには、対応するCapabilityを持っている必要があります。
ネットワーク、入力デバイス、GPU、プロセス操作、システム情報の取得なども、必要に応じてCapabilityによって制御されます。

Capabilityを用いることで、単純なユーザーIDや管理者権限だけに依存しない、より細かい権限制御が可能になります。

mnuでは、次のような方針でCapabilityを扱います。

- 権限は暗黙ではなく明示的に与える
- 不要なCapabilityを持つプロセスを作らない
- サービスごとに必要最小限の権限だけを与える
- カーネルは重要な操作の前にCapabilityを検査する
- 権限の付与や分類は専用サービスで管理できるようにする
- 物理メモリの参照や物理ページのマッピングは `memory.phys.map` / `memory.phys.translate` で分離する
- Service権限だけで物理アドレス取得や物理ページ mapping を許可しない
- パスレベルの権限は `capabilities.manage` を持つ core.service 系の起動経路を起点に、カーネル初期登録で管理する
- 実際のパス定義はカーネルが検証し、path registry を参照して判定する

この設計により、仮にサービスの一部が侵害された場合でも、被害範囲をそのサービスが持つCapabilityの範囲に抑えることを目指します。

## メモリ管理

mnuカーネルは、各プロセスに独立した仮想アドレス空間を提供します。
プロセスは自分に割り当てられたメモリ領域のみを操作でき、他のプロセスやカーネルメモリへ直接アクセスすることはできません。

メモリ管理では、次の要素を扱います。

- 物理フレームの管理
- 仮想アドレス空間の管理
- ページテーブルの構築
- ユーザー空間とカーネル空間の分離
- ヒープ領域の管理
- メモリマップ領域の管理
- ユーザーポインタの検証

ユーザー空間から渡されるポインタは信頼しません。
システムコールやIPCでユーザーポインタを扱う場合、カーネルは対象アドレスが本当にユーザー空間に属しているか、要求されたアクセス権を満たしているかを検査します。

この検査を徹底することで、ユーザープロセスからカーネルメモリを破壊したり、他プロセスのメモリを不正に読み書きしたりすることを防ぎます。

## プロセス管理

mnuのプロセスは、独立したアドレス空間と実行コンテキストを持ちます。
カーネルはプロセスの生成、終了、スケジューリング、状態遷移を管理します。

プロセスには役割に応じた権限レベルがあります。
一般アプリケーションは最小限の権限で動作し、システムサービスやコアサービスは必要なCapabilityを持って動作します。

プロセス管理では、次の状態を扱います。

- 実行可能
- 実行中
- 待機中
- IPC待ち
- 終了済み
- 隔離済み

異常な状態になったプロセスは、必要に応じて隔離されます。
隔離されたプロセスは通常のスケジューリング対象から外され、システム全体への影響を抑えます。

## 割り込みと例外処理

カーネルはCPU例外、ハードウェア割り込み、システムコール入口を管理します。
ページフォルト、不正命令、一般保護例外などのCPU例外は、発生元のコンテキストに応じて処理されます。

ユーザープロセスで発生した例外は、原則としてそのプロセスの異常として扱います。
カーネル内部で発生した例外は、より重大な障害として扱われます。

割り込み処理では、ハードウェアからの通知を受け取り、必要に応じて対応するサービスやドライバへイベントを配送します。
可能な限り、複雑な処理を割り込みハンドラ内で直接行わず、後続の処理へ委譲します。

## ドライバモデル

mnuでは、ドライバを可能な限りユーザー空間で動作させる方針を取ります。
ドライバはデバイスに対応するサービスとして動作し、アプリケーションや他のサービスからの要求をIPCで受け取ります。

ユーザー空間ドライバには、対象デバイスへアクセスするためのCapabilityが与えられます。
これにより、ドライバが扱えるデバイスや操作範囲を制限できます。

この設計には、次の利点があります。

- ドライバのクラッシュがカーネル全体を巻き込みにくい
- ドライバを個別に更新しやすい
- デバイスごとの権限を分離しやすい
- カーネル内に大量のドライバコードを持たずに済む

ただし、すべての処理をユーザー空間だけで完結できるわけではありません。
割り込み、MMIO、DMA、初期化に関わる一部の処理では、カーネルによる低レベルな支援が必要になります。
mnuカーネルは、そのための最小限の機構を提供します。

ユーザー空間で動作するドライバについては、[PlugKitドキュメント](./plugkit/about.md)でさらに詳しく説明されています。

カーネル空間で動作するドライバについては、[Cextドキュメント](./cext.md)でさらに詳しく説明されています。

## セキュリティ

mnuのセキュリティ設計では、プロセスやサービスを最初から信頼しないことを前提にします。
ユーザー空間から渡される値、ポインタ、メッセージ、Capability要求はすべて検査対象です。

特に重要な方針は次のとおりです。

- カーネルメモリをユーザー空間から直接参照させない
- ユーザーポインタを必ず検証する
- 権限のないシステムコールを拒否する
- サービスごとにCapabilityを分離する
- 他プロセスの物理アドレスやページテーブル操作は、対象プロセスへのアクセス権と専用Capabilityの両方を要求する
- 破損したIPC状態を隔離する
- カーネル内部の不正状態を検出しやすくする
- 障害発生時に被害範囲を限定する

mnuは、単に機能が動くことだけではなく、不正な入力や壊れた状態でも安全側に倒れることを重視します。
これは、長期的に安定したOSを作るための基本方針です。

## ブートと初期化

mnuの起動時には、ブートローダから渡された情報をもとに、カーネルが最低限の初期化を行います。
メモリマップ、フレームバッファ、initfs、rootfsなどの情報を受け取り、カーネル内部の管理構造を構築します。

初期化では、主に次の処理を行います。

- コンソールの初期化
- 物理メモリ管理の初期化
- 仮想メモリ管理の初期化
- 割り込み処理の初期化
- システムコールの初期化
- 初期ファイルシステムの認識
- 最初のサービスの起動
- スケジューラの開始

カーネルは起動後、必要なサービスを順に起動し、OSとして利用できる状態へ移行します。
以降の高レベルな機能は、ユーザー空間サービスによって提供されます。

ブートローダーが実装するべき機能や、カーネルが起動時に期待する情報の詳細については、[ブートドキュメント](./boot.md)
でさらに詳しく説明されています。

## 目標

mnuカーネルの最終的な目標は、小さく、安全で、高速なマイクロカーネルを基盤として、実用的なOSを構築することです。

単に既存OSの構造を模倣するのではなく、Capability、IPC、ユーザー空間サービス、障害隔離を中心に据えた設計を採用します。
これにより、クラッシュしにくく、拡張しやすく、理解しやすいOSを目指します。

mnuは実験的なOSでありながら、設計段階から実用性を意識しています。
カーネル本体は最小限に保ち、OSの機能はサービスとして組み立てることで、柔軟で保守しやすいシステムを目指します。
